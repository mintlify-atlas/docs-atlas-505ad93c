---
title: "Quickstart"
description: "Download your first SRA accession and convert it to FASTQ"
---

This guide walks you through a complete end-to-end workflow: configuring the toolkit, downloading an SRA accession, and converting it to FASTQ. The example uses accession `SRR000001`, a small public dataset, but the same steps apply to any accession.

<Note>
  SRA data is available in two formats. **SRA Normalized Format** (`.sra`) contains full per-base quality scores. **SRA Lite** (`.sralite`) uses simplified quality scores (Q30 for reads that pass the filter, Q3 for rejected reads), and is substantially smaller. If your downstream analysis does not depend on precise quality values, you can request SRA Lite files in `vdb-config` to reduce download times and disk usage.
</Note>

<Steps>
  <Step title="Configure the toolkit">
    Before downloading any data, set a local cache directory where prefetched files will be stored. The interactive configuration tool lets you review and adjust all toolkit settings:

    ```bash
    vdb-config --interactive
    ```

    To set only the cache directory non-interactively — which is useful for scripted or headless environments — run:

    ```bash
    vdb-config --set /repository/user/main/public/root=$HOME/sra-cache
    ```

    This creates (or designates) `~/sra-cache` as the root of your local SRA cache. All subsequent `prefetch` downloads will land here.

    <Tip>
      If you are running on a system with a large, fast local disk mounted at a non-default path (such as `/scratch` on an HPC cluster), point the cache there rather than to your home directory to avoid filling your quota and to benefit from higher I/O throughput.
    </Tip>
  </Step>

  <Step title="Prefetch the accession">
    Download the SRA accession and any required reference sequences to your local cache:

    ```bash
    prefetch SRR000001
    ```

    Expected output:

    ```text
    2026-04-09T10:12:03 prefetch.3.4.1: Downloading 'SRR000001'...
    2026-04-09T10:12:03 prefetch.3.4.1:  Downloading via HTTPS...
    2026-04-09T10:12:05 prefetch.3.4.1:  HTTPS download succeed
    2026-04-09T10:12:05 prefetch.3.4.1: 'SRR000001' was downloaded successfully
    ```

    The accession file is saved under your configured cache directory at `~/sra-cache/sra/SRR000001.sra`.

    You can prefetch multiple accessions in a single call:

    ```bash
    prefetch SRR000001 SRR000002 SRR000003
    ```
  </Step>

  <Step title="Convert to FASTQ with fasterq-dump">
    Convert the downloaded accession to FASTQ format:

    ```bash
    fasterq-dump SRR000001
    ```

    `fasterq-dump` automatically detects whether the data is single-end or paired-end and writes the appropriate output files.

    Expected output (single-end data):

    ```text
    spots read      : 1,000
    reads read      : 1,000
    reads written   : 1,000
    ```

    The resulting file(s) will appear in your current working directory:

    - **Single-end**: `SRR000001.fastq`
    - **Paired-end**: `SRR000001_1.fastq` and `SRR000001_2.fastq`

    <Tip>
      By default, `fasterq-dump` writes temporary files to a directory called `fast.tmp` in the current working directory. If that filesystem is small or slow, redirect temp files to a faster location — for example, a RAM-backed filesystem:

      ```bash
      fasterq-dump -t /dev/shm SRR000001
      ```

      Using `/dev/shm` can significantly reduce wall-clock time for large accessions on systems where RAM is plentiful. The scratch directory must have roughly as much free space as the final output.
    </Tip>
  </Step>

  <Step title="Inspect the output">
    Confirm the files were created and check the first few reads:

    ```bash
    ls -lh SRR000001*.fastq
    ```

    Example output:

    ```text
    -rw-r--r-- 1 user user 142K Apr  9 10:12 SRR000001.fastq
    ```

    Preview the first two reads (8 lines of FASTQ):

    ```bash
    head -8 SRR000001.fastq
    ```

    Example output:

    ```text
    @SRR000001.1 NCBI-SRA-ReadGroup:SRR000001 length=36
    AAGCTTGCATGCCTGCAGGTCGACTCTAGAGGATCC
    +SRR000001.1 NCBI-SRA-ReadGroup:SRR000001 length=36
    IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
    @SRR000001.2 NCBI-SRA-ReadGroup:SRR000001 length=36
    AAGCTTGCATGCCTGCAGGTCGACTCTAGAGGATCC
    +SRR000001.2 NCBI-SRA-ReadGroup:SRR000001 length=36
    IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
    ```
  </Step>

  <Step title="Stream without downloading (optional)">
    For small accessions or exploratory work, you can stream reads directly to stdout without writing an intermediate `.sra` file to disk:

    ```bash
    fasterq-dump --stdout SRR000001 | head -40
    ```

    This pipes the FASTQ output directly to the next command in your pipeline, which is useful for quick quality checks or when disk space is limited.
  </Step>
</Steps>

## Working with paired-end data

Paired-end accessions require you to choose how the two read mates are written. `fasterq-dump` provides three modes:

| Flag | Behavior |
|------|----------|
| `--split-files` | Writes mate 1 to `_1.fastq` and mate 2 to `_2.fastq`. Unpaired reads are discarded. |
| `--split-3` | Same as `--split-files`, but saves any unpaired reads to a third file with no suffix (e.g. `SRR000001.fastq`). |
| *(default)* | Writes all reads interleaved into a single file. Mate pairs appear consecutively. |

For most paired-end workflows, `--split-files` is the appropriate choice:

```bash
fasterq-dump --split-files SRR000002
```

Expected output files:

```text
SRR000002_1.fastq   # mate 1
SRR000002_2.fastq   # mate 2
```

Use `--split-3` when the accession may contain a mix of paired and unpaired reads (common with some older datasets):

```bash
fasterq-dump --split-3 SRR000002
```

This produces up to three files:

```text
SRR000002.fastq     # unpaired reads (if any)
SRR000002_1.fastq   # mate 1
SRR000002_2.fastq   # mate 2
```

<Note>
  If you use `--include-technical` to include technical reads in your output, `fasterq-dump` automatically switches to `--split-files` mode regardless of what you specify. Technical reads and biological reads are written to separate files.
</Note>
